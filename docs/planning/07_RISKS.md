# 07 — 风险登记簿

> 日期：2026-03-02
> 每条风险包括：描述、类别、发生概率、影响程度、缓解方案、应急预案

## R01: App Store 审核拒绝（键盘扩展 + Open Access）

| 属性 | 详情 |
|------|------|
| **描述** | 键盘扩展请求 Open Access（完全访问）会触发 Apple 的额外隐私审核。审核团队可能认为语音输入键盘不需要完全访问权限，或者对音频数据发送到第三方服务器有顾虑 |
| **类别** | 审核 |
| **概率** | 中 |
| **影响** | 高——键盘扩展是 iOS 版核心价值，审核被拒意味着无法通过 App Store 分发完整功能 |
| **缓解方案** | 1) 审核附注详细说明 Open Access 仅用于联网调用语音识别 API，不做 keystroke logging。2) 准备操作演示视频（录屏完整使用流程）。3) 隐私标签准确声明数据收集范围。4) App 内提供清晰的隐私说明页。5) 研究同类 App（如搜狗输入法、百度输入法）的审核先例 |
| **应急预案** | 1) 根据审核反馈修改并重新提交。2) 初期通过 TestFlight 分发（最多 10,000 外部测试者），绕过正式审核。3) 极端情况下降级为不需要 Open Access 的纯离线模式（Apple Speech，但精度大幅下降） |

## R02: App Store 审核拒绝（麦克风权限在键盘扩展中使用）

| 属性 | 详情 |
|------|------|
| **描述** | Apple 可能认为键盘扩展使用麦克风权限存在隐私风险。虽然技术上允许（iOS 系统支持键盘扩展录音），但审核团队可能要求更多解释 |
| **类别** | 审核 |
| **概率** | 中 |
| **影响** | 高——麦克风是核心功能，被拒等于键盘扩展无法工作 |
| **缓解方案** | 1) NSMicrophoneUsageDescription 清晰说明用途。2) 键盘扩展中录音仅在用户主动按住按钮时启动（不做被动监听）。3) 录音结束后立即释放 AVAudioSession。4) 引用系统键盘自带的听写功能作为类比论据 |
| **应急预案** | 同 R01 的应急预案 |

## R03: ASR API 依赖风险（服务变更/成本/可用性）

| 属性 | 详情 |
|------|------|
| **描述** | 核心 ASR 服务（Qwen3-ASR-Flash）由阿里云提供。服务商可能调整价格、变更 API 接口、限流、停服、或在特定地区不可用 |
| **类别** | 依赖 |
| **概率** | 低 |
| **影响** | 高——ASR 是核心 pipeline 的第一步，不可用等于整个 App 失效 |
| **缓解方案** | 1) 支持多 ASR provider（Qwen + Whisper API），用户可切换。2) P1 集成 Apple Speech 作为离线降级。3) 使用 ASRProvider 协议抽象，添加新 provider 成本低。4) 监控 DashScope API 的状态和变更通知 |
| **应急预案** | 1) ASR 失败时自动降级到 Apple Speech。2) 保留音频文件让用户稍后重试。3) 如果 Qwen 长期不可用，快速接入其他云端 ASR（Groq Whisper、DeepGram 等） |

## R04: iOS 键盘扩展内存超限被系统 kill

| 属性 | 详情 |
|------|------|
| **描述** | iOS 键盘扩展有严格的内存限制（约 48-80MB，因设备和 iOS 版本不同）。如果 ASR 网络请求的 response 数据较大，或 SwiftUI 视图层级复杂，可能触发系统 OOM kill |
| **类别** | 技术 |
| **概率** | 中 |
| **影响** | 高——被 kill 会导致键盘突然消失，用户需要重新切换键盘 |
| **缓解方案** | 1) 只用云端 ASR（网络请求内存开销 ~5MB），不加载本地模型。2) SwiftUI 视图保持简单（避免复杂动画和大图片）。3) 使用 Instruments Memory Graph 进行性能分析。4) 音频文件及时释放（处理后立即删除）。5) Autorelease pool 管理临时对象 |
| **应急预案** | 1) 键盘扩展被 kill 后用户重新激活时自动恢复状态。2) 如果频繁被 kill，在主 App 中显示提示让用户使用主 App 模式 |

## R05: 后台权限被苹果拒绝

| 属性 | 详情 |
|------|------|
| **描述** | 如果未来需要 Background Audio 权限（如 Live Activity 配合后台录音），App Store 审核可能拒绝。Apple 对后台音频权限的审核标准要求 App 有充分的音频播放/录制理由 |
| **类别** | 审核 |
| **概率** | 高（如果申请后台权限） |
| **影响** | 中——当前设计不依赖后台权限，键盘扩展天然前台 |
| **缓解方案** | 1) 不申请 UIBackgroundModes: audio。2) 键盘扩展作为当前输入法时天然处于活跃状态，不需要后台权限。3) 主 App 只在前台录音 |
| **应急预案** | 维持"不使用后台权限"的策略。如果 P2 阶段需要 Live Activity + 录音，仅在灵动岛显示状态，录音仍在前台进行 |

## R06: 竞品风险

| 属性 | 详情 |
|------|------|
| **描述** | iOS 上已有多款语音输入类 App 和输入法（搜狗输入法语音、讯飞输入法、Apple 原生听写、Whisper Transcription 等 App）。Vox 可能面临市场竞争 |
| **类别** | 商业 |
| **概率** | 高（竞品已存在） |
| **影响** | 中——Vox 有差异化定位（BYOK、中英混合、LLM 后处理、隐私） |
| **缓解方案** | 1) 差异化定位：BYOK 模式（无订阅费、无数据收集）+ LLM 后处理（去口语化）+ 中英混合优化。2) 强调隐私：用户自己的 API Key，数据不经过 Vox 服务器。3) 面向技术用户群体（开发者、写作者）。4) 与 macOS 版联动形成生态 |
| **应急预案** | 如果市场竞争激烈，聚焦小众但忠诚的用户群体（BYOK + 隐私意识强的技术用户） |

## R07: Qwen ASR 在弱网/海外延迟高或不可达

| 属性 | 详情 |
|------|------|
| **描述** | DashScope 服务器在中国大陆（dashscope.aliyuncs.com）。问题不仅是延迟：1) 海外用户可能面临 DNS 解析失败、TLS 握手超时、链路抖动等导致服务**完全不可用**（而非仅仅慢）；2) 弱网（4G/3G）下 base64 音频上传延迟可能达 5-10 秒；3) 特定地区的网络策略可能阻断对阿里云域名的访问 |
| **类别** | 技术 |
| **概率** | 高（移动网络不稳定是常态；海外访问阿里云域名本身有不确定性） |
| **影响** | 高——不仅影响体验，海外场景下可能导致核心功能完全不可用 |
| **缓解方案** | 1) P2 引入音频压缩减少上传体积（需验证 Opus 可行性）。2) 设置合理超时（25s）+ 重试（2 次）。3) 弱网/请求中显示明确的进度提示。4) **推荐海外用户使用 Whisper API**（Groq 等全球分布的服务商）作为替代——这是 BYOK 模式下最有效的"多区域"策略。5) 在首次设置引导中根据用户地区推荐合适的 ASR provider |
| **应急预案** | 1) 自动切换到 Apple Speech（精度降级但本地执行，零网络依赖）。2) 保留音频文件让用户网络恢复后重试。3) 长期观察各 provider 的全球可用性数据 |

## R08: AVAudioSession 与宿主 App 冲突

| 属性 | 详情 |
|------|------|
| **描述** | 键盘扩展启动录音时的 AVAudioSession 配置可能与宿主 App 的音频产生冲突。某些 App（如音乐播放器、视频通话）可能不正确响应 .mixWithOthers 选项 |
| **类别** | 技术 |
| **概率** | 中 |
| **影响** | 中——影响特定场景的用户体验（如边听音乐边用 Vox 输入） |
| **缓解方案** | 1) 使用 .playAndRecord + .mixWithOthers + .defaultToSpeaker + .allowBluetooth 的最佳实践配置。2) 录音结束后使用 .notifyOthersOnDeactivation 通知宿主 App 恢复音频。3) 广泛测试常见宿主 App（微信、备忘录、Safari、Apple Music、Spotify） |
| **应急预案** | 如果特定 App 存在不兼容，在 FAQ / 帮助中说明已知问题 |

## R09: 用户不愿手动添加键盘

| 属性 | 详情 |
|------|------|
| **描述** | iOS 添加第三方键盘需要 Settings → Keyboards → Add New Keyboard → Vox → Allow Full Access，步骤较多。用户可能因为摩擦成本放弃使用键盘扩展 |
| **类别** | 商业 |
| **概率** | 高 |
| **影响** | 中——影响键盘扩展的采纳率，但不影响主 App 功能 |
| **缓解方案** | 1) 首次使用引导中提供清晰的**图文步骤**指导（截图标注每一步操作）。2) 使用 TipKit（iOS 17+）在适当时机提示用户添加键盘。3) 跳转到 App 自身设置页面（`UIApplication.openSettingsURLString` 只能跳到本 App 设置页，**无法精确跳到键盘设置**），配合图文引导用户手动导航到 Settings → General → Keyboard → Keyboards。4) 主 App 提供独立的录音→剪贴板功能作为键盘扩展的替代 |
| **应急预案** | 如果键盘扩展采纳率低，增强主 App 的独立使用体验（Siri Shortcut、Action Button 等替代触发方式） |

## R10: LLM Provider API 不稳定

| 属性 | 详情 |
|------|------|
| **描述** | LLM 后处理依赖第三方 API（Kimi/Qwen-LLM/DeepSeek 等），这些服务可能限流、变更接口、调整定价 |
| **类别** | 依赖 |
| **概率** | 中 |
| **影响** | 低——LLM 后处理是可选功能，失败自动降级到 ASR 原文 |
| **缓解方案** | 1) LLM 失败时无感降级到原文 + TextFormatter。2) 支持多 provider 切换。3) 超时设置为 12s（不让用户等太久）。4) 用户可以关闭 LLM 后处理功能 |
| **应急预案** | 向用户推荐仍可用的 provider，或关闭后处理 |

## R11: Apple 系统更新导致 API 变更

| 属性 | 详情 |
|------|------|
| **描述** | Apple 每年 iOS 大版本更新可能带来键盘扩展 API 变更、新的权限限制、内存限制调整等 |
| **类别** | 技术 |
| **概率** | 中（每年一次系统更新） |
| **影响** | 中——可能需要代码适配 |
| **缓解方案** | 1) 关注 WWDC 和 iOS Beta 版本变更日志。2) 在 iOS Beta 期间提前测试兼容性。3) 使用稳定的系统 API（避免私有 API）。4) 最低版本设为 iOS 17，覆盖充分 |
| **应急预案** | 及时跟进适配，发布更新版本 |

## R12: 项目工期超出预期

| 属性 | 详情 |
|------|------|
| **描述** | 键盘扩展开发的复杂度可能超出预期（调试困难、边缘情况多、不同设备表现不一致），导致项目延期 |
| **类别** | 时间 |
| **概率** | 中 |
| **影响** | 中——影响发布节奏但不影响技术可行性 |
| **缓解方案** | 1) P0 MVP 先发 TestFlight，不等键盘扩展完成。2) 严格的 Sprint 验收标准，每个 Sprint 有可回滚点。3) 分批发布：v0.1.0（主 App）→ v0.2.0（+键盘）→ v0.3.0（+增强）。4) 工时估算已包含 30% 缓冲 |
| **应急预案** | 调整 P1 范围，将翻译模式/历史记录推迟到 P2 |

## R13: Bundle ID / App 名称冲突

| 属性 | 详情 |
|------|------|
| **描述** | "Vox" 是一个常见名称，App Store 中可能已有同名 App。Bundle ID 如 com.xxx.vox 也可能与已注册的 ID 冲突 |
| **类别** | 商业 |
| **概率** | 中 |
| **影响** | 低——改名即可解决 |
| **缓解方案** | 1) 提前在 App Store Connect 检查 App 名称可用性。2) 准备备用名称（如 "Vox Input"、"VoxVoice"）。3) Bundle ID 使用个人域名前缀确保唯一性 |
| **应急预案** | 使用备用名称 |

## R14: App Group 容器内日志/历史文本泄露

| 属性 | 详情 |
|------|------|
| **描述** | debug.log 和 history.json 位于 App Group 共享容器中，可能包含用户的识别文本、LLM 响应、错误上下文等敏感信息。在键盘扩展场景下（用户输入可能涉及密码、私密对话、医疗信息），日志泄露面尤其需要关注。加上"键盘+语音"组合，审核敏感度较高 |
| **类别** | 安全/合规 |
| **概率** | 中 |
| **影响** | 中——可能触发审核关注，且实际隐私风险存在 |
| **缓解方案** | 1) Release 版本默认关闭详细日志（logLevel = .error）。2) 日志中对 ASR/LLM 文本内容只记录摘要（前 20 字符 + "..."）。3) 日志按天滚动，保留 7 天自动清理。4) 导出日志功能需用户显式确认。5) 任何日志级别都不记录 API Key 明文。详见 04_ARCHITECTURE.md 日志安全策略 |
| **应急预案** | 审核如果质疑日志内容，提供代码级证据说明生产版本不记录敏感文本 |

## R15: 隐私合规风险（BYOK 模式的合规边界）

| 属性 | 详情 |
|------|------|
| **描述** | Vox 上传音频到用户自有的 ASR API 账号。虽然 BYOK 模式让 Vox 本身不做数据中转/存储，但仍需准确告知用户数据流向。海外用户使用中国大陆的 Qwen ASR 涉及跨境传输。审核团队可能要求更详细的隐私说明 |
| **类别** | 合规 |
| **概率** | 中 |
| **影响** | 中——可能延迟 App Store 上架 |
| **缓解方案** | 1) 准备清晰的隐私政策，说明 Vox 是工具而非数据处理者。2) App 内提供数据流说明页面。3) 隐私标签准确声明。4) 如果未来引入服务端代理层，合规要求会显著升级——届时需要 DPA、数据处理协议等。详见 03_REQUIREMENTS.md 隐私合规准备清单 |
| **应急预案** | 根据审核反馈补充隐私文档并重新提交 |
