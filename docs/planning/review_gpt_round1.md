## 问题清单

### R001
- **问题类别**：错误判断
- **位置**：`04_ARCHITECTURE.md` → 1.整体架构（App Group 桥梁）、5.2 数据存储分布；`02_IOS_CHALLENGES.md` 挑战2 解决方案表
- **问题说明**：文档多处把 **Keychain 共享** 与 **App Group** 混为同一机制（例如“Keychain + accessGroup = App Group 级共享”）。这在 iOS 安全模型上不准确：
  - App Group 解决的是共享容器（文件/UserDefaults）
  - Keychain 共享依赖 **Keychain Sharing entitlement**（`keychain-access-groups`），且 access group 需符合 Team ID 规则
- **为什么是问题**：如果按当前文档实施，极可能出现“主 App 能写 key，键盘扩展读不到 key”的致命问题，直接阻塞 P1。
- **建议补充**：在架构与 requirements 中明确拆分两套能力：
  1) App Group: `com.apple.security.application-groups`
  2) Keychain Sharing: `keychain-access-groups`
- **严重程度**：**高**

### R002
- **问题类别**：遗漏挑战
- **位置**：`02_IOS_CHALLENGES.md` 挑战5（键盘扩展限制）
- **问题说明**：未覆盖一个关键平台限制：**自定义键盘在密码框/安全输入框/某些受限输入场景会被系统禁止**，且部分 App 可主动拒绝第三方键盘。
- **为什么是问题**：这会直接影响“全场景可用性”预期，若不在 PRD 里提前说明，用户会误认为功能失效。
- **建议补充**：加入“不可用场景矩阵”（secureTextEntry、银行卡/密码输入、宿主禁用 extension point 等）及 UX 降级提示。
- **严重程度**：**高**

### R003
- **问题类别**：遗漏 Requirement
- **位置**：`03_REQUIREMENTS.md`（未体现），`04_ARCHITECTURE.md` 键盘 UI 设计部分（未体现）
- **问题说明**：缺少 iOS 自定义键盘审核硬性要求：**必须提供切换到下一输入法（地球键）能力**，并正确处理 `needsInputModeSwitchKey` / input mode list。
- **为什么是问题**：不满足该要求会触发审核风险，且实际可用性差。
- **建议补充**：在 requirements 与 Sprint 4 验收里新增“地球键行为 + 审核合规检查”。
- **严重程度**：**高**

### R004
- **问题类别**：遗漏挑战
- **位置**：`04_ARCHITECTURE.md`（键盘扩展采用 SwiftUI + UIHostingController）
- **问题说明**：未充分讨论 **UIKit vs SwiftUI 在键盘扩展内的性能/内存/首帧延迟权衡**。目前只有“可行方案”，没有“成本边界”。
- **为什么是问题**：Keyboard Extension 对冷启动和内存极敏感，复杂 SwiftUI 层级可能比主 App 更容易触发卡顿和 kill。
- **建议补充**：给出明确预算（如首帧<300ms、峰值内存<55MB）及 fallback（关键路径回退 UIKit）。
- **严重程度**：**中**

### R005
- **问题类别**：错误判断
- **位置**：`07_RISKS.md` → R09 缓解方案（“深链接到系统键盘页面”）
- **问题说明**：文档写到可“深链接到键盘设置页”，该能力在 iOS 实践中通常不可精确跳转，`openSettingsURLString`一般只能到本 App 设置页。
- **为什么是问题**：会导致交互方案落空，用户引导路径与实现不一致。
- **建议补充**：改为“跳转 App 设置 + 图文步骤引导用户手动进入键盘设置”。
- **严重程度**：**中**

### R006
- **问题类别**：遗漏风险
- **位置**：`08_OPEN_QUESTIONS.md` Q03；`06_SPRINT_PLAN.md` 发布策略
- **问题说明**：多处表达“TestFlight 可回避审核压力”，但 **外部测试** 仍需 Beta App Review。
- **为什么是问题**：容易低估时间成本与审核阻塞点，影响版本计划。
- **建议补充**：区分“内部测试无需审核”与“外部测试需审核”，并把首次外测审核时间纳入计划缓冲。
- **严重程度**：**中**

### R007
- **问题类别**：遗漏 Requirement
- **位置**：`03_REQUIREMENTS.md`（App Store 准备项不完整）
- **问题说明**：缺少强制准备项：**隐私政策 URL / 支持 URL / 数据处理说明页面**（App Store Connect 元数据层面）。
- **为什么是问题**：即使代码完备，也会在提审阶段卡住。
- **建议补充**：在“前置依赖检查清单”中新增法律与元数据资产：Privacy Policy、Support URL、联系人邮箱、数据处理说明。
- **严重程度**：**高**

### R008
- **问题类别**：遗漏 Requirement
- **位置**：`03_REQUIREMENTS.md`、`07_RISKS.md`
- **问题说明**：提到隐私标签，但未落到 **GDPR / 中国个人信息保护（PIPL）/ 跨境传输** 的具体准备动作（告知、同意、撤回、删除、保留周期）。
- **为什么是问题**：该项目会上传语音数据到第三方 ASR/LLM，属于高敏感合规面。
- **建议补充**：新增“合规准备清单”：数据流图、最小化原则、保留与删除策略、用户同意文案、第三方 DPA/条款映射。
- **严重程度**：**高**

### R009
- **问题类别**：遗漏风险
- **位置**：`04_ARCHITECTURE.md` → 5.2/5.3（debug.log/history/prompt 存储）；`07_RISKS.md`（未覆盖）
- **问题说明**：未显式评估 **App Group 容器内日志/历史文本泄露面**，尤其 debug.log 可能包含识别文本、错误上下文、provider 响应片段。
- **为什么是问题**：这会形成隐私与审计风险，且与“键盘+语音”组合叠加后审核敏感度更高。
- **建议补充**：日志脱敏、默认关闭详细日志、按天滚动与自动清理、导出需用户显式确认。
- **严重程度**：**高**

### R010
- **问题类别**：错误判断
- **位置**：`04_ARCHITECTURE.md` 技术栈表（音频压缩 PCM→Opus）；`06_SPRINT_PLAN.md` P2 音频压缩任务
- **问题说明**：把“AVAudioConverter 直接转 Opus”写成默认路径，技术可行性表述过于确定。iOS 上 Opus 编码链路并非所有版本/封装都可直接落地，常见可行稳态是 AAC/CAF。
- **为什么是问题**：若按既定假设排期，可能在 P2 发生实现偏差与工时失真。
- **建议补充**：将其改为“待验证决策项”，并准备 fallback：AAC-LC（m4a）或保持 PCM+限长策略。
- **严重程度**：**中**

### R011
- **问题类别**：遗漏挑战
- **位置**：`02_IOS_CHALLENGES.md` 挑战3/5；`04_ARCHITECTURE.md` 键盘流程
- **问题说明**：虽然讨论了“不要后台录音”，但没展开 **扩展进程生命周期短、长请求易被挂起/回收** 的约束（这与后台模式不同，是 extension 自身生命周期问题）。
- **为什么是问题**：会影响网络超时、重试策略与 UX（用户等待时键盘消失/重启）。
- **建议补充**：为键盘扩展单独定义“短事务策略”：录音时长上限、请求上限、失败即时回退主 App。
- **严重程度**：**中**

### R012
- **问题类别**：遗漏开放问题
- **位置**：`08_OPEN_QUESTIONS.md`（缺失）
- **问题说明**：缺失关键决策：**是否引入服务端代理层**（API Key 托管、审计、风控、区域路由、Provider 切换）。
- **为什么是问题**：当前全 BYOK+端侧直连在 keyboard extension 场景下有明显 UX 与安全边界问题；至少需要明确“不做后端”的长期代价。
- **建议补充**：新增 open question：纯 BYOK vs 轻代理（token exchange）架构取舍。
- **严重程度**：**高**

### R013
- **问题类别**：遗漏开放问题
- **位置**：`08_OPEN_QUESTIONS.md`（缺失）；`02_IOS_CHALLENGES.md`（未系统比较）
- **问题说明**：缺少 **Share Extension / Action Extension / Keyboard Extension** 三种入口的系统化权衡与阶段策略。
- **为什么是问题**：不同入口在权限、可达性、审核风险、用户教育成本差异很大，影响 roadmap。
- **建议补充**：新增决策表（触发成本、注入能力、审核风险、实现复杂度）。
- **严重程度**：**中**

### R014
- **问题类别**：遗漏开放问题
- **位置**：`08_OPEN_QUESTIONS.md`（缺失）；`06_SPRINT_PLAN.md` P2（Live Activity）
- **问题说明**：文档列了 Live Activity，但未明确 **是否由主 App 发起/维护、与键盘扩展协同是否可行**。
- **为什么是问题**：该功能跨进程状态同步复杂，若边界不清会形成“计划功能不可落地”。
- **建议补充**：新增决策项：Live Activity 是否仅用于主 App 会话；键盘扩展只做本地 UI 状态，不做实时岛状态承诺。
- **严重程度**：**中**

### R015
- **问题类别**：遗漏 Requirement
- **位置**：`03_REQUIREMENTS.md` → 1.Apple Developer 账号
- **问题说明**：账号类型说明过于简化（“个人足够”），缺少组织账号在品牌、协作、法务主体、后续转移成本上的影响评估；也未明确企业账号不适用于公开 App Store 分发。

### R016
- **问题类别**：遗漏风险
- **位置**：`07_RISKS.md` R07（仅讨论延迟）
- **问题说明**：已提海外延迟，但未覆盖 **境外可达性/区域网络策略** 风险（DNS、链路抖动、TLS 握手失败、特定地区访问策略）。
- **为什么是问题**：对“海外用户可用性”不仅是慢的问题，可能是不可用。
- **建议补充**：新增风险条目：多区域 endpoint、provider fallback、地域化默认配置。
- **严重程度**：**中**

### R017
- **问题类别**：遗漏挑战
- **位置**：`02_IOS_CHALLENGES.md`（未展开）；`04_ARCHITECTURE.md` 2.2 录音层（固定 16kHz）
- **问题说明**：对 **音频采样率/路由兼容** 描述不足。iPhone 内建麦克风、蓝牙设备（尤其 SCO）实际采样率与通道常变化，直接固定 16kHz 需验证转换链路稳定性。
- **为什么是问题**：可能导致特定耳机/车载场景下识别异常、音质劣化或失败。
- **建议补充**：加入设备矩阵与路由测试（内建麦克风、AirPods、蓝牙车机），并在 AudioSession 里明确 route change handling。
- **严重程度**：**中**
